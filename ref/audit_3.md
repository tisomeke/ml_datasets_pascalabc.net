## 2. it_salary — Средний датасет

### Audit Summary
Проверены: 4 sample-программы, 3 meta, 3 CSV, 4 отчёта.

### Critical Issues
1. **OrdinalEncoder для city** — в `03_baseline_model.pas` и `04_preprocessing_raw.pas` все категориальные признаки кодируются через `OrdinalEncoder`. Для `grade` (intern→junior→middle→senior→lead) это осмысленно. Для `city` (moscow→spb→regions) — нет: код навязывает порядок, которого не существует. Для KNN это не критично (расстояния всё равно считаются по всем координатам), но методологически неверно. **Рекомендация:** добавить комментарий, что `OrdinalEncoder` выбран для компактности, а в продакшене стоит использовать `OneHotEncoder` для неупорядоченных категорий.

2. **04_preprocessing_raw.pas — posted_date удалён без демонстрации извлечения признаков.** Программа называется `preprocessing_raw`, но единственный "сырой" признак `posted_date` (DateTime) просто дропается. Ожидалось бы извлечение месяца или квартала через `WithColumnInt`. **Рекомендация:** добавить извлечение `posted_month` как демонстрацию работы с DateTime.

### Improvements
1. **Z-score для выбросов** — корректно: `Filter(r -> Abs(r['salary_rub'].ToFloat - salMean) <= 3 * salStd)`. Порог 3σ — стандартный. Хорошо.
2. **Imputer стратегии** — медиана для числовых, константа для категориальных. Верный выбор.
3. **seed = 42** — фиксирован везде. Воспроизводимость обеспечена.

### Verdict
⚠️ **Требует изменений.** Issue #2 (posted_date) — добавить извлечение признака. Issue #1 — добавить комментарий-дисклеймер.

## 3. data_air_cities — Большой датасет

### Audit Summary
Проверены: 4 sample-программы, 2 meta, 2 CSV, 4 отчёта, scripts/preprocess_data.py, ТЗ.

### Critical Issues

#### 🔴 CRITICAL #1: Комментарий про -1.0 вводит в заблуждение

**Где:** `03_baseline_model.pas` (строки 8–10) и `04_preprocessing_raw.pas` (строки 7–9).

**Что написано:**
```
// В данных Росгидромета пропуски закодированы как -1.0 (скрытые пропуски).
// Imputer заменяет только явные NA, поэтому -1.0 остаются как есть.
// Для baseline это приемлемо — модель учится игнорировать -1.0 как значения.
```

**Почему это неверно:** KNN — алгоритм, основанный на расстояниях. Значение `-1.0` не игнорируется моделью, а интерпретируется как реальное числовое значение. После `StandardScaler` `-1.0` превращается в некоторое отрицательное число. Точки со значением `-1.0` по данному признаку образуют ложный кластер "пропущенных" наблюдений, что искажает расстояния до ближайших соседей. Модель НЕ учится игнорировать `-1.0` — она учится использовать `-1.0` как маркер пропуска, что не переносится на новые данные.

**Влияние:** R² = 0.68 может быть завышен или искажён из-за того, что модель использует `-1.0` как информативный признак (наличие/отсутствие данных о выбросах), а не как пропуск.

**Рекомендация:** 
- Заменить `-1.0` на `NaN` через `WithColumnFloat` перед `Imputer`
- Или явно отфильтровать строки с `-1.0` в признаках (как уже сделано для target)
- Комментарий исправить на честный: "Внимание: -1.0 искажает расстояния в KNN. В реальном проекте требуется замена на NA."

#### 🔴 CRITICAL #2: Дублирование признаков в 04_preprocessing_raw.pas

**Где:** `04_preprocessing_raw.pas`, строка 50.

**Что происходит:** Кодируется `region` через `OrdinalEncoder`, но в матрицу признаков также включается `region_oktmo`. Поскольку `region` и `region_oktmo` находятся во взаимно-однозначном соответствии (77 регионов ↔ 77 кодов ОКТМО), это дублирование информации. Мультиколлинеарности для KNN нет, но признак бесполезен и размывает важность других признаков.

**Рекомендация:** Убрать `region_oktmo` из `featureCols` или закодировать `region` через `region_oktmo` (взять первые 2 цифры как федеральный округ).

#### 🟡 WARNING #3: Несбалансированность классов не учтена

**Где:** `quality_report.md` указано, что класс 4 — 10.7%, но в sample-программах нет демонстрации взвешенных метрик. Для учебного датасета это допустимо, но стоило бы отметить в коде.

### Improvements
1. **sanpin_era** — корректно реализован через `WithColumnInt`. Хороший feature engineering.
2. **Фильтрация target** — `df.Filter(row -> row[targetCol].IsValid and (row[targetCol] <> -1.0))` — правильно.
3. **Разделение кода** — 03 (baseline, минимум обработки) и 04 (с предобработкой) логично разделены.
4. **ТЗ** — адаптировано под MLABC API, все ссылки валидны.

### Verdict
🔴 **Требует изменений.** Critical #1 и #2 должны быть исправлены перед фиксацией.
